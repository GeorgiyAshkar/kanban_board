# Детальный code review и UX-аудит приложения **Personal Kanban Board**

Дата аудита: **27 апреля 2026**.  
Основание: анализ backend/frontend кода, конфигурации запуска и доступных тестов в репозитории.

---

## 1) Executive summary (коротко)

Приложение уже выглядит как зрелый **MVP+**: функционально покрывает полный цикл задач (доска, история, комментарии, чек-листы, теги, напоминания, аналитика, архив, backup/import), а не только «создать/удалить карточку». В архитектуре есть задел на эксплуатацию (health/ready/metrics/alerts), optimistic locking, и интеграционные тесты на ключевые сценарии.

Главный вывод: продукт уже ценен, но его «предел роста» упирается не в отсутствие фич, а в **операционную надежность и архитектурную управляемость**:

1. Метрики подключены частично (endpoint есть, но фактический сбор HTTP-метрик не встроен в request pipeline).
2. Фоновый worker напоминаний живет внутри процесса API (риск дублей/гонок при масштабировании в несколько инстансов).
3. Фронтенд-оркестрация перегружена (крупный controller-hook + дублирующиеся hooks), что ухудшает скорость безопасных изменений.
4. UX на критичных действиях часто «best effort»: локальные optimistic update без явного UX для конфликтов, ошибок, и фоновых состояний.

## 2) Сильные стороны приложения (с обоснованием)

### 2.1. Сильная предметная модель и хорошая функциональная полнота

В модели есть не только `tasks`, но и связанные контуры: `task_history`, `task_comments`, `task_reminders`, `reminder_notifications`, `task_checklist_items`, `tags`, `task_tags`, `board_columns`. Это важный признак масштабируемости по функциям: можно наращивать аналитические и процессные сценарии без «пересборки базы».  

### 2.2. Backend закрывает полный жизненный цикл задач

Есть CRUD и операционные endpoints (`move`, `archive`, `restore`, `complete`), фильтрация для board-view, агрегаты (`/today`, `/analytics/report`), детализация задачи (`/tasks/{id}/details`) и backup/import. Это снижает зависимость UI от «ручной склейки» нескольких API и делает архитектуру чище.

### 2.3. Реализован optimistic locking на обновлении задачи

`PATCH /tasks/{id}` требует `row_version`, возвращает `428`, если версии нет, и `409`, если версия устарела. Это правильный шаг для защиты от silent overwrite и обязательный фундамент для командной работы и будущей realtime-коллаборации.

### 2.4. История изменений ведется системно

Многие операции логируются в `task_history` (создание, перемещение, смена статуса/приоритета, архив/восстановление, авто-эскалации и т.п.). Это улучшает аудитируемость, расследование инцидентов и UX-прозрачность («что произошло с задачей»).

### 2.5. Есть интеграционные тесты на бизнес-критичный поток

В тестах покрыты сценарии lifecycle, комментарии/чек-листы/теги, board/details агрегация, уведомления, фильтрация, аналитика и optimistic locking. Это хороший уровень «контрактной защиты» против регрессий.

## 3) Слабые зоны и узкие места (с подробным разбором)

## 3.1. Наблюдаемость реализована не до конца (операционная слепая зона)

**Что хорошо:** есть `ops`-эндпоинты `/health`, `/ready`, `/metrics`, `/alerts` и отдельный registry метрик.

**Проблема:** счетчики/гистограммы HTTP не обновляются через middleware/интерцептор запросов. В коде есть `observe_request`, но фактических вызовов в request pipeline нет.

**Риск:** метрики latency/error-rate в runtime могут оставаться нулевыми/неконсистентными, а алерты — срабатывать некорректно. При росте нагрузки это критично.

## 3.2. Фоновая обработка напоминаний встроена прямо в API-процесс

`ReminderNotificationWorker` стартует на `startup` приложения и запускает бесконечный цикл внутри процесса сервера.

**Риск масштабирования:** при запуске нескольких backend-процессов/реплик каждый поднимет собственный worker. Это увеличивает риск дублей/гонок/непредсказуемой нагрузки, особенно при переносе на Kubernetes/горизонтальный scaling.

**Риск надежности:** при перезапуске API фоновая обработка тоже перезапускается; lifecycle задачи и lifecycle worker жестко связаны.

## 3.3. Смешение статуса и колонок (дублирование источников истины)

У задачи одновременно живут `status` и `board_column_id`, при этом при move/complete статус синхронизируется с `canonical_status` колонки.

**Риск:** два поля описывают одно и то же бизнес-состояние на разных уровнях абстракции. При расширении домена (кастомные workflow, подтипы статусов, роль-правила) высок риск рассинхронизации и сложных edge-case.

## 3.4. UX обработки ошибок и состояний загрузки неравномерен

В `App`/страницах часть экранов просто рендерит данные без явных error-state и retry-state. Например, `HistoryPage` ожидает массив и не показывает «ошибка загрузки/пустое состояние как сценарий», `TodayPage` показывает только «Загрузка...», без разбора ошибок сети.

**Риск UX:** пользователь получает «тишину» или неполную обратную связь при сетевых сбоях/таймаутах. Для рабочего инструмента это заметно бьет по доверию.

## 3.5. Kanban drag-and-drop ограничен (масштаб UX по доске)

В board используется базовый HTML5 DnD: задача переносится в колонку по drop, позиция устанавливается фактически в конец (`targetSize`).

**Риски UX/масштаба:**
- нет полноценного reorder внутри колонки;
- слабее поддержка touch/mobile UX;
- меньше предсказуемость при больших колонках.

Для масштабируемой канбан-доски это один из ключевых UX-узлов.

## 3.6. Перегрузка orchestration-слоя во frontend + дублирующиеся hooks

`useBoardController` агрегирует query-состояния, действия, saved filters, модальные состояния, analytics-параметры, optimistic flows и т.д. Параллельно присутствуют отдельные `useBoardQueries` и `useTaskMutations`, а также неиспользуемый `useReminderNotifications`.

**Риск:** рост когнитивной сложности, более высокая стоимость изменений, сложнее делать feature isolation и безопасный рефакторинг.

## 3.7. Локальные prompt-диалоги на ключевых сценариях редактирования

В Drawer для части операций (например, создание фильтра/тега, редактирование чек-листа) используется `window.prompt`.

**UX-проблема:**
- не единый визуальный паттерн;
- нет валидации/подсказок в форме;
- хуже accessibility и контроль ошибок;
- сложно масштабировать сценарии (цвет, описание, предпросмотр, подсказки и т.д.).

## 3.8. Ограничение board-загрузки на фронте «в лоб» (limit=200)

`fetchBoardData` запрашивает `limit: 200` и `offset: 0`. Для раннего этапа это нормально, но при росте числа задач UX и производительность начнут упираться в отсутствие виртуализации/пагинации/инкрементальной подгрузки.

## 4) Что перерабатывать в первую очередь (приоритизация)

## P0 — обязательно до активного масштабирования

### P0.1. Довести observability до production-уровня

**Сделать:**
- добавить HTTP middleware, который на каждый запрос вызывает `metrics_registry.observe_request`;
- добавить request-id/correlation-id и базовый structured logging для route-level событий;
- проверить корректность `/metrics` в реальной нагрузке (не только «endpoint существует»).

**Почему первым:** без измеримости невозможно управлять деградациями, SLA и capacity.

### P0.2. Вынести reminder worker в отдельный process/worker-role

**Сделать:**
- отделить worker от API-процесса (например, отдельный контейнер role);
- гарантировать single-consumer semantics (блокировки, leader election или queue-механика);
- в API оставить только read/write endpoints + ack.

**Почему первым:** это ключ к горизонтальному scaling без дублей и race-condition.

### P0.3. Единый UX-контур ошибок/загрузки/конфликтов данных

**Сделать:**
- унифицировать error states для всех страниц;
- добавить retry и человекочитаемые сообщения об ошибках API;
- отдельный UX для optimistic-lock conflict (`409`): «данные устарели, перезагрузить и сравнить изменения».

**Почему первым:** резко повышает пользовательское доверие и снижает «невидимые» потери времени команды.

## P1 — следующий этап (после стабилизации)

### P1.1. Рефакторинг frontend в feature-модули

**Сделать:**
- разделить controller по bounded-context: board, task-details, reports, settings;
- удалить/свести дубли (`useBoardQueries`, `useTaskMutations`, `useReminderNotifications`) в единую стратегию;
- формализовать слой API + слой use-cases + presentation-компоненты.

### P1.2. Перевести доску на полнофункциональный DnD + reorder

**Сделать:**
- внедрить `dnd-kit` сценарии: reorder внутри колонки, keyboard/touch accessibility, drag overlay;
- хранить и синхронизировать позиции безопасно и предсказуемо.

### P1.3. Убрать `window.prompt` в пользу модальных/inline-форм

**Сделать:**
- компактные controlled-формы с валидацией, подсказками, явными confirm/cancel;
- единый дизайн-паттерн UX на всей доске.

## P2 — точки роста после архитектурной стабилизации

1. Серверная пагинация + клиентская виртуализация карточек.
2. Событийная модель для интеграций (webhook/event bus) и advanced automation.
3. Multi-user/security контур (users, roles, workspace scopes, audit trail actor).
4. Расширенная аналитика (aging WIP, throughput variability, SLA breach trends).

## 5) Сильные/слабые зоны по UX отдельно

### Сильные UX-зоны

- Единая рабочая модель «доска + детали задачи + контекстные вкладки».
- Наличие screen-специализаций: Today / History / Archive / Reports / Settings.
- Быстрые операции в карточках (чек-лист, теги, архивирование).

### Слабые UX-зоны

- Непоследовательная обратная связь по ошибкам и фоновой загрузке.
- Ограниченный DnD-поток (нет reorder и richer drag behavior).
- Нативные prompt-диалоги вместо управляемых форм.
- Ограниченная масштабируемость интерфейса при больших объемах карточек.

---

## 6) Карта рисков (если ничего не менять)

1. **Операционный риск:** инциденты «видны поздно» из-за неполной телеметрии.
2. **Риск масштабирования:** дубли фоновых обработчиков при multiple replicas.
3. **Риск скорости разработки:** рост времени изменений из-за перегруженного orchestration-слоя.
4. **UX-риск:** при росте команды пользователи будут упираться в конфликты данных и слабую обратную связь ошибок.

## 7) Практический план на 6–8 недель

### Спринт 1–2
- Внедрить request-metrics middleware + унифицированные error responses.
- Ввести UX-компоненты для loading/error/empty состояний.
- Прототип conflict-resolution для `409 row_version`.

### Спринт 3–4
- Декомпозировать frontend controller по feature-модулям.
- Заменить prompt-сценарии на модальные/inline-формы.
- Подготовить API/схему для reorder внутри колонки.

### Спринт 5–6
- Вынести reminder worker в отдельный runtime-role.
- Добавить защиту от дублей обработки (single-run semantics).
- Запустить нагрузочную проверку `/metrics` + уведомлений + board-потока.

## 8) Вывод

Приложение находится в хорошем состоянии для этапа MVP+: сильный доменный каркас и много полезной функциональности уже реализованы. Главный приоритет дальнейшего роста — не «добавить еще экранов», а повысить системную зрелость: **наблюдаемость, разделение runtime-ролей, предсказуемый UX ошибок/конфликтов и упрощение frontend-архитектуры**. После этих шагов расширение функционала (командный режим, richer analytics, automation) станет существенно дешевле и безопаснее.
