# Аудит приложения Personal Kanban Board

Дата аудита: **26 апреля 2026**.
Основание: анализ текущего кода backend/frontend, конфигурации запуска и тестов.

## 1) Сильные стороны (с обоснованием)

1. **Широкое покрытие предметной области в модели данных.**
   В модели уже есть задачи, история, комментарии, чек-листы, напоминания, теги, колонки и события уведомлений. Это снижает стоимость масштабирования функционала (аналитика, SLA, автоматизации). 

2. **Хороший API-охват жизненного цикла задач.**
   Реализованы CRUD и операционные действия (`move/archive/restore/complete`), плюс агрегированные представления (`/tasks/board`, `/today`, `/analytics/report`), что дает не только «хранение», но и рабочие процессы.

3. **Realtime-уведомления уже сделаны через SSE, а не только pull.**
   В backend есть `/notifications/stream` с `text/event-stream`, а frontend подписывается через `EventSource` и подтверждает доставку события ack-запросом.

4. **Есть готовые эксплуатационные заделы.**
   Присутствуют endpoints для readiness/metrics/alerts и минимальная метрика латентности/ошибок, а также Docker Compose c dual-mode (SQLite + PostgreSQL профиль).

5. **Неплохая база тестов на критические сценарии.**
   Интеграционные тесты покрывают жизненный цикл задачи, фильтрацию, напоминания, историю, аналитику и автоматизацию дедлайнов.

---

## 2) Слабые стороны и риски (с обоснованием)

1. **Критичный технический разрыв: ops-роутер не подключен в приложении.**
   В `backend/app/api/ops.py` есть `/ready`, `/metrics`, `/alerts`, но в `main.py` этот роутер не `include_router`.
   **Риск:** observability-функции «существуют в коде, но недоступны».

2. **Перегруженный orchestration-уровень во frontend (`App.tsx`).**
   В одном компоненте собраны маршрутизация экранов, query-слой, SSE-логика, мутации и часть бизнес-правил.
   **Риск:** медленное развитие UI, высокий риск регрессий при изменениях.

3. **Нет multi-user security-уровня (authn/authz).**
   Модель и API ориентированы на single-user/local сценарий: нет сущности пользователя, ролей и scoped-доступа.
   **Риск:** невозможность безопасного командного использования.

4. **Нет явного контроля конкуренции обновлений задач.**
   В API используется обычный PATCH без versioning/etag/optimistic locking.
   **Риск:** «последняя запись победила» при параллельном редактировании.

5. **Сильная зависимость от SQLite по умолчанию.**
   PostgreSQL-профиль есть, но основная тропа и dev-путь завязаны на SQLite.
   **Риск:** деградация при росте нагрузки/конкурентных операций.

## 3) Потенциальные точки роста

### P0 (максимальный эффект)

1. **Подключить и стандартизировать ops endpoints.**
   - Добавить `app.include_router(ops.router)`.
   - Зафиксировать `/health`, `/ready`, `/metrics` как эксплуатационный контракт.
   **Почему:** даст измеримость качества и деградаций сразу.

2. **Декомпозировать `App.tsx` на feature-level hooks/контейнеры.**
   - Вынести orchestration в `useBoardController`, `useNotifications`, `useTaskDetails`.
   - Оставить `App.tsx` как композиционный слой.
   **Почему:** уменьшит связанность и ускорит релизы.

3. **Закрыть «базу безопасности» для multi-user режима.**
   - JWT/session + user model + ownership/role checks.
   **Почему:** без этого нельзя безопасно расти в командный сценарий.

4. **Надежность редактирования: optimistic locking.**
   - Версионное поле у задач (`row_version`) и проверка в PATCH.
   **Почему:** устранит silent overwrite.

### P1 (стратегически важно)

1. **Функциональные WIP-лимиты колонок + SLA-дедлайны.**
   **Почему:** переводит доску из «списка карточек» в инструмент управления потоком.

2. **Расширение отчётов до управленческого контура.**
   - Burnup/Burndown, aging WIP, throughput variability.
   **Почему:** помогает принимать решения, а не только смотреть «сколько задач».

3. **Импорт/экспорт и резервные сценарии восстановления.**
   **Почему:** снижает операционный риск и повышает доверие к системе.

## 4) Что реализовать/добавить/модернизировать

### Реализовать (новое)

1. **Пользователи, роли, рабочие пространства (workspace/team).**
   Обоснование: текущая архитектура богата функциями задач, но без границ доступа.

2. **Recurring tasks + шаблоны задач.**
   Обоснование: высокая ценность для регулярных процессов (рутинные циклы).

3. **Mass actions (bulk edit / bulk move / bulk archive).**
   Обоснование: критично для еженедельной гигиены бэклога.

### Добавить (к текущему UX)

1. **Undo/redo на рискованные действия.**
   Обоснование: снижает цену пользовательской ошибки.

2. **Saved Views 2.0 (не только фильтры, но и layout/сортировки/группировка).**
   Обоснование: ускоряет повседневные ритуалы планирования.

3. **Сигналы перегрузки колонки (WIP сигнал + рекомендации).**
   Обоснование: прямое влияние на lead time и просрочки.

### Модернизировать (существующее)

1. **Frontend state-management (anti-god-component рефакторинг).**
2. **Ops/observability слой: подключение роутера + унификация метрик.**
3. **PATCH-сценарии: добавить версионирование для предотвращения конфликтов.**
4. **По умолчанию готовить PostgreSQL-путь как production baseline.**

## 5) Рекомендуемый roadmap (12 недель)

- **Недели 1–3:** ops router wiring, декомпозиция `App.tsx`, технический долг по типизации/мутациям.
- **Недели 4–7:** auth + роли + ownership, optimistic locking в задачах.
- **Недели 8–12:** WIP/SLA, расширенная аналитика, импорт/экспорт и безопасное восстановление.

---

## 6) KPI эффекта

1. API 5xx rate, p95 latency.
2. Notification delivery lag (создание события → ack).
3. Median task lead time / cycle time.
4. Доля просроченных задач.
5. Частота конфликтов редактирования (после ввода optimistic locking).

---

## 7) Вывод

У приложения уже сильная функциональная база и качественный MVP+ каркас. Следующий шаг — усилить **управляемость (ops), масштабируемость (архитектура frontend/backend) и безопасность (multi-user)**. Это даст переход от «хорошего локального продукта» к устойчивой production-системе.
