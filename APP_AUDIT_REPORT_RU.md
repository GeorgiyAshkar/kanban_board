# Аудит приложения Personal Kanban Board

Дата аудита: 2026-04-26  
Контекст: анализ текущего состояния MVP+ на основе исходного кода и README.

## 1) Краткий профиль продукта

Приложение уже покрывает ключевой операционный цикл личного/командного трекера задач:
- канбан-доска с колонками;
- карточка задачи с комментариями, чек-листом, тегами, историей;
- архив/восстановление;
- раздел «Сегодня»;
- базовые напоминания и браузерные уведомления.

Это хороший фундамент для перехода от MVP к рабочей production-версии.

---

## 2) Сильные стороны (и почему это важно)

### 2.1. Хорошо продуманная предметная модель
**Что вижу:** в backend-моделях есть не только `tasks`, но и связанные сущности: `task_history`, `task_comments`, `task_reminders`, `task_checklist_items`, `tags`, `task_tags`, `board_columns`, а также таблица notification-событий.  
**Почему это плюс:** у вас изначально заложена расширяемая модель, поэтому дальнейшие фичи (аналитика, SLA, отчеты, интеграции) можно добавлять без «ломки» архитектуры.

### 2.2. История изменений как first-class функциональность
**Что вижу:** изменения задач логируются централизованно через сервис истории и специальные action-типы.  
**Почему это плюс:** это повышает доверие к системе, дает базу для аудита, аналитики и «объяснимости» (кто/что/когда изменил).

### 2.3. Backend уже поддерживает важные пользовательские сценарии
**Что вижу:** есть маршруты для CRUD задач, move/archive/restore/complete, чек-листа, комментариев, тегов, колонок и агрегаций (`/today`, board-view).  
**Почему это плюс:** пользователь получает законченные сценарии «создал → уточнил → переместил → завершил/архивировал».

### 2.4. Frontend уже ориентирован на UX, а не только на функциональность
**Что вижу:** есть отдельные экраны (Board/Today/History/Archive/Settings), модалка создания задачи, drawer-детализация, адаптивность, drag-and-drop, оптимистичные апдейты отдельных операций.  
**Почему это плюс:** продукт воспринимается как «живое приложение», а не «форма над БД».

### 2.5. Покрытие тестами ключевых API-потоков уже есть
**Что вижу:** smoke и integration тесты проверяют жизненный цикл задачи, историю, чек-лист/комментарии/теги, агрегаты доски и доставку reminder-событий.  
**Почему это плюс:** это снижает риск регрессий при доработке.

### 2.6. Есть контейнеризация и понятный путь деплоя
**Что вижу:** docker-compose, Dockerfile для frontend/backend, скрипт упаковки образов.  
**Почему это плюс:** легче переносить решение между средами и готовить поставку «в коробке».

## 3) Слабые стороны и риски (с обоснованием)

### 3.1. Монолитный `App.tsx` и высокая связность UI-логики
**Проблема:** главный компонент содержит много бизнес-логики, запросов, оптимистичных мутаций и побочных эффектов.  
**Риск:** сложнее сопровождать, выше шанс регрессий, сложнее онбординг новых разработчиков.

### 3.2. Отсутствует явная аутентификация/авторизация
**Проблема:** API ориентирован на локальный режим без user/session модели.  
**Риск:** при выходе за рамки single-user локального сценария нельзя безопасно разделять данные и права.

### 3.3. SQLite как единственная БД для текущей реализации
**Проблема:** для MVP нормально, но при многопользовательской и конкурентной нагрузке SQLite быстро становится узким местом (блокировки, ограничения по масштабированию).  
**Риск:** деградация UX при росте объема данных и числа операций.

### 3.4. Ограниченная observability
**Проблема:** нет явного слоя метрик/трейсинга/структурированных логов на уровне продукта.  
**Риск:** сложно диагностировать «медленные» запросы, отказы, качество уведомлений и SLA.

### 3.5. Часть возможностей реализована частично
**Проблема:** в README прямо отмечены незавершенные пункты (dnd-kit reorder, расширенное редактирование полей, фильтры по ТЗ, импорт/экспорт, аналитика).  
**Риск:** продукт может «застрять» между MVP и production-ready без приоритизированного roadmap.

### 3.6. Браузерные уведомления — polling, а не push
**Проблема:** frontend опрашивает события с интервалом, а не использует веб-сокеты/SSE.  
**Риск:** лишняя нагрузка, задержка доставки, менее предсказуемое поведение на слабых сетях.

## 4) Точки роста и развития (приоритет + ценность)

### P0 — высокий бизнес-эффект / низкий-умеренный риск

#### 4.1. Разгрузить frontend-слой состояния
**Что сделать:** вынести мутации/селекторы из `App.tsx` в feature-hooks (`useTaskMutations`, `useBoardQueries`, `useReminderNotifications`) и модульные сервисы.  
**Зачем:** уменьшение сложности, рост скорости разработки, меньше случайных side-effect конфликтов.

#### 4.2. Довести UX редактирования задачи до «полного профиля»
**Что сделать:** единая форма редактирования всех полей задачи (включая оценку, исполнителя, цвет, emoji, даты, проект).  
**Зачем:** пользователь перестает «допридумывать» процесс во внешних инструментах.

#### 4.3. Расширить фильтрацию и поиск
**Что сделать:** фильтры по тегам, датам, исполнителю, колонкам, completed/archived, быстрые сохраненные фильтры.  
**Зачем:** без этого доска быстро теряет управляемость при росте числа задач.

#### 4.4. Импорт/экспорт/резервное копирование
**Что сделать:** JSON/CSV экспорт задач, архив бэкапа с метаданными, импорт с валидацией схемы и dry-run.  
**Зачем:** снимает страх потери данных и упрощает миграцию между окружениями.

### P1 — стратегический эффект

#### 4.5. Миграция на PostgreSQL (опционально dual-mode)
**Что сделать:** добавить профиль запуска с Postgres + миграции + индексы под основные сценарии выборок.  
**Зачем:** подготовка к росту, конкурентной нагрузке, аналитическим запросам.

#### 4.6. Настоящий realtime для напоминаний
**Что сделать:** заменить polling на SSE/WebSocket канал уведомлений.  
**Зачем:** ниже задержка, меньше лишних запросов, лучше UX.

#### 4.7. Наблюдаемость и эксплуатация
**Что сделать:** structured logging, health+readiness, базовые метрики (latency, error rate, queue size reminders), алерты.  
**Зачем:** продукт становится управляемым в продакшене.

### P2 — дифференцирующие возможности

#### 4.8. Аналитика и отчеты
**Что сделать:** cycle time, lead time, WIP, просрочки, velocity по периодам, диаграммы трендов.  
**Зачем:** превращает трекер задач в инструмент принятия решений.

#### 4.9. Автоматизации
**Что сделать:** правила вида “если deadline < 24h и статус != done — поднять приоритет + создать напоминание”.  
**Зачем:** снижает ручную операционку и повышает предсказуемость исполнения.

## 5) Функционал, который стоит реализовать/добавить/модернизировать

### 5.1. Реализовать (новый)
1. **Saved Views** (сохраненные фильтры + сортировки + группировки).  
   Обоснование: ускоряет работу power-user и повторяемые ритуалы планирования.
2. **Recurring tasks (шаблоны задач)**.  
   Обоснование: регулярные процессы сейчас приходится заводить вручную.
3. **Bulk operations** (массовое перемещение, теги, архив).  
   Обоснование: критично при еженедельной гигиене бэклога.
4. **Экспорт/импорт + backup scheduler**.  
   Обоснование: управляемость жизненного цикла данных.

### 5.2. Добавить (в текущий UX)
1. **Глобальные hotkeys** (создать задачу, поиск, переместить).  
   Обоснование: снижает стоимость рутинных действий.
2. **Undo для рискованных операций** (архив/удаление/массовые изменения).  
   Обоснование: уменьшает цену пользовательской ошибки.
3. **Inline-редактирование карточки на доске**.  
   Обоснование: меньше переключений в drawer/modal.
4. **WIP-лимиты колонок**.  
   Обоснование: дисциплина потока и раннее обнаружение перегруза.

### 5.3. Модернизировать (существующее)
1. **Drag-and-drop на dnd-kit с reorder внутри и между колонками**.  
   Обоснование: это базовое ожидание для зрелого kanban UX.
2. **Систему уведомлений (polling -> realtime)**.  
   Обоснование: выше оперативность и ниже фоновая нагрузка.
3. **Слой API/Query-мутаций на frontend** (единые optimistic helpers + rollback strategy).  
   Обоснование: уменьшается дублирование и сложность.
4. **Поисковый индекс и SQL-индексы** по часто используемым фильтрам.  
   Обоснование: стабильная производительность при росте данных.

## 6) Рекомендуемый план внедрения (90 дней)

### Этап 1 (1–3 недели)
- Рефакторинг frontend state-management (декомпозиция `App.tsx`).
- Полная форма редактирования задачи.
- Расширенные фильтры + saved views (минимальная версия).

### Этап 2 (4–7 недель)
- dnd-kit reorder.
- Массовые операции + undo.
- Экспорт/импорт (JSON) + ручной backup/restore.

### Этап 3 (8–12 недель)
- Realtime уведомления (SSE/WebSocket).
- Метрики и эксплуатационные дашборды.
- Базовый модуль аналитики (lead/cycle time, WIP, просрочки).

## 7) KPI, по которым стоит оценивать эффект изменений

1. **Time-to-update-task** (время на изменение задачи) — должно снижаться.
2. **Task throughput/week** — должно расти при стабильном качестве.
3. **Overdue ratio** — доля просроченных задач должна снижаться.
4. **Notification delivery lag** — задержка доставки напоминаний должна снижаться.
5. **UI error rate / API error rate** — должно снижаться после рефакторинга.

## 8) Итог

У приложения уже сильный «скелет»: зрелая модель данных, рабочие end-to-end пользовательские сценарии, история изменений, хорошая база UI/UX и контейнеризация.  
Главный следующий шаг — **перейти от функционального MVP к управляемой, масштабируемой product-архитектуре**: декомпозиция frontend-слоя, углубление workflow-функций, улучшение observability и подготовка к росту данных/нагрузки.
