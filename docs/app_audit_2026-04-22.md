# Аудит приложения Kanban Board (22 апреля 2026)

## 1) Краткий контекст

Приложение уже покрывает широкий MVP+/MMP набор:
- канбан-доска с колонками и перемещением задач;
- карточка задачи с редактированием, комментариями, тегами, чек-листом, историей;
- разделы «Сегодня», «История», «Архив», «Настройки»;
- backend на FastAPI + SQLAlchemy + Alembic, frontend на React + TypeScript + Vite.

Это означает, что продукт находится на хорошей стадии «рабочий personal/team task manager», и дальнейший рост — не в «добавить любую фичу», а в системном усилении UX, производительности, качества данных и сценариев коллаборации.

---

## 2) Сильные стороны

### 2.1. Хорошее функциональное покрытие MVP
**Почему это важно:** уже есть основа для ежедневной эксплуатации, а не только демо.

**Подтверждение в проекте:**
- реализованы задачи, история, комментарии, чек-лист, напоминания, теги, колонки и агрегатор «Сегодня»;
- frontend уже имеет отдельные экраны Board/Today/History/Archive/Settings.

**Эффект:** быстрее получать обратную связь от пользователей и приоритизировать развитие на данных.

### 2.2. Архитектурный фундамент для развития
**Почему это важно:** наличие миграций и разделения на API/сервисы/модели снижает стоимость будущих изменений.

**Подтверждение в проекте:**
- Alembic миграции + запуск миграций на старте;
- backend выделяет сервисный слой (`services/tasks.py`), где инкапсулировано логирование изменений.

**Эффект:** проще эволюционировать схему БД и правила доменной логики.

### 2.3. Полезная доменная модель задач
**Почему это важно:** заранее заложены поля, необходимые для перехода от «личных заметок» к более формальному task tracking.

**Подтверждение:**
- в модели есть дедлайн, planned return, done-флаги, оценка/факт времени, поля исполнителя, архивность.

**Эффект:** можно быстро нарастить отчеты и SLA/контроль исполнения без переработки ядра.

### 2.4. Внятный UX-скелет на фронте
**Почему это важно:** пользователь уже может выполнять основной цикл «создал → продвинул → прокомментировал → завершил/архивировал».

**Подтверждение:**
- BoardView + TaskDrawer и интерактивные операции (чек-лист, теги, комментарии, архив, редактирование полей);
- есть optimistic updates для части действий.

**Эффект:** ощущение «живого» приложения даже при неидеальной сети.

## 3) Слабые стороны и риски

### 3.1. Отсутствие аутентификации/авторизации и мультипользовательской изоляции
**Риск:** для реального командного режима и production-развертывания это критический блокер (безопасность и разграничение доступа).

**Что видно:**
- нет явных auth-эндпоинтов, роли/права не прослеживаются;
- в комментариях/истории используется `local_user`.

**Последствия:** невозможность безопасно масштабировать продукт для нескольких пользователей/команд.

### 3.2. Ограниченное тестовое покрытие
**Риск:** регрессии при росте кода.

**Что видно:**
- есть только smoke-тесты на health и листинг задач;
- отсутствуют интеграционные кейсы по критичным операциям (move/archive/restore/tags/checklist/history).

**Последствия:** рост цены каждого релиза и снижение доверия к изменениям.

### 3.3. Потенциальные проблемы производительности при масштабировании
**Риск:** деградация UX и рост нагрузки на API.

**Что видно:**
- на фронте много независимых запросов и массовых invalidation;
- board metadata грузится отдельным вызовом по ids, плюс отдельные task-specific queries;
- backend список задач без серверной нормальной пагинации по умолчанию (limit опционален).

**Последствия:** на сотнях/тысячах задач интерфейс может стать тяжелым.

### 3.4. Уязвимость к рассинхронизации статуса и колонки
**Риск:** нарушение целостности бизнес-логики.

**Что видно:**
- в UI статус при DnD вычисляется эвристикой по названию колонки (`includes('вход')`, `includes('выполн')` и т.д.).

**Последствия:** при переименовании колонок логика может «ломаться» (неверный `status`).

### 3.5. «Сегодня» работает на UTC и без пользовательской таймзоны
**Риск:** ошибки попадания задач в блоки today/overdue вокруг полуночи.

**Что видно:**
- `today` рассчитывается через `datetime.utcnow()`.

**Последствия:** для пользователей из разных TZ блоки «на сегодня» могут восприниматься как некорректные.

### 3.6. Dev hygiene: versioned node_modules / нестабильное окружение
**Риск:** шумные диффы, нестабильные сборки и конфликты в репозитории.

**Что видно:**
- в репозитории присутствует `frontend/node_modules`;
- локальный запуск `pytest` падает из-за отсутствующего `httpx` в окружении.

**Последствия:** сложно поддерживать воспроизводимость CI/CD и ревью.

## 4) Точки роста и развития

### Трек A (краткосрочный, 1–3 недели): «Надежность и управляемость»
1. **Усилить тестовый контур:**
   - backend: интеграционные тесты по критическим сценариям;
   - frontend: smoke E2E (Playwright/Cypress) на базовый user flow.
2. **Привести dev-окружение к воспроизводимости:**
   - зафиксировать недостающие dev-зависимости (`httpx` для тестов),
   - удалить `node_modules` из VCS и полагаться на lockfile.
3. **Нормализовать маппинг колонка ↔ статус:**
   - хранить canonical status в колонке (на backend), не вычислять эвристикой на UI.

**Зачем:** минимизирует технический риск и ускоряет цикл релизов.

### Трек B (среднесрочный, 1–2 месяца): «Продуктовый UX + масштабирование»
1. **Серверные фильтры + полнотекстовый поиск + пагинация по умолчанию.**
2. **Сжатие количества запросов на Board:**
   - агрегированный endpoint для карточки/доски,
   - аккуратнее invalidation в react-query.
3. **Системные уведомления и reliable reminders:**
   - браузерные уведомления + очередь/воркер для напоминаний.

**Зачем:** улучшает отзывчивость и ежедневную ценность приложения.

### Трек C (стратегический, 2–4 месяца): «Team-ready платформа»
1. **Auth + RBAC + tenancy.**
2. **Совместная работа (mentions, assignment workflow, activity subscriptions).**
3. **Отчеты/аналитика (lead time, throughput, aging, SLA breaches).**

**Зачем:** открывает путь к B2B/team-сценариям и монетизируемым планам.

## 5) Что реализовать / добавить / модернизировать (приоритизированный backlog)

## P0 — обязательно

1. **Auth (JWT/session) + пользовательский контекст в истории/комментариях**
   - **Обоснование:** без этого нельзя безопасно масштабировать продукт и доверять журналу действий.

2. **RBAC и доступ к данным по owner/workspace**
   - **Обоснование:** защищает данные и делает приложение пригодным для команды.

3. **Расширить автотесты (API + критические UI-флоу)**
   - **Обоснование:** снижает вероятность регрессий на ключевых операциях (move/archive/edit/checklist/tags).

4. **Стабилизировать статусную модель**
   - **Обоснование:** убрать эвристику по названию колонки; бизнес-логика должна быть декларативной.

5. **Time zone strategy для Today/Deadline**
   - **Обоснование:** корректность «дневных» блоков — core value ежедневного планирования.

## P1 — высокий приоритет

6. **Расширенные фильтры и сохраненные представления (saved views)**
   - **Обоснование:** рост продуктивности при увеличении числа задач.

7. **Recurring tasks + recurring reminders**
   - **Обоснование:** сильно увеличивает retention для регулярных процессов.

8. **Подзадачи и зависимости задач**
   - **Обоснование:** поддержка более сложных сценариев планирования.

9. **Сокращение количества API round-trips на Board**
   - **Обоснование:** лучшее время отклика и ощущение «легкости» интерфейса.

10. **Импорт/экспорт/backup (JSON/CSV)**
    - **Обоснование:** снижает vendor lock-in и повышает доверие к инструменту.

## P2 — следующий этап

11. **Уведомления (browser/push/email) + digest**
12. **Отчеты по потоку задач (cycle time, aging, completion trends)**
13. **WebSocket/live-updates для командной синхронизации**
14. **Шаблоны задач/чек-листов и автоматизации (правила)**
15. **Мобильные UX-оптимизации и офлайн-режим**

## 7) Измеримые KPI развития

- **Техкачество:**
  - покрытие критических API-сценариев ≥ 70%;
  - доля failed deploys < 5%.
- **UX/скорость:**
  - p95 загрузки board < 1.5s на 1000 задач;
  - p95 действия move/save < 300ms (optimistic feedback).
- **Продукт:**
  - WAU/MAU, retention D30;
  - % задач с дедлайном и % выполненных в срок;
  - доля пользователей, использующих reminders/tags/checklists.

---

## 8) Главный вывод

У приложения уже сильный функциональный фундамент и правильный технологический вектор. Наиболее выгодный следующий шаг — не «хаотично добавить фичи», а пройти три оси зрелости: **(1) безопасность и многопользовательский контур, (2) надежность через тесты и корректность доменной логики, (3) производительность и умный UX для масштаба**. После этого можно уверенно развивать team-сценарии и аналитику.
